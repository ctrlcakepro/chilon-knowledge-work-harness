# 演示交付协议

目标：把 PPT、HTML deck、课堂展示、组会汇报、答辩演示统一按“演讲交付物”处理，而不是按普通页面或排版任务处理。

## 先判断这是不是演讲任务
如果用户提到以下任一信号，默认按演讲交付协议执行：
- PPT
- slides
- deck
- HTML 幻灯片
- 组会汇报
- 课堂展示
- 分享稿
- 演讲稿
- 答辩
- 明天要讲
- 提词器
- speaker notes
- presenter view

## 演讲任务的核心不是页面，而是讲述控制
默认目标不是“做得像网页”，而是同时满足：
- 有清楚主判断
- 有可讲述的页序
- 有稳定翻页
- 可全屏
- 有 notes / presenter mode / 提词支持
- 观众视图与讲者支持分离

## 默认工作流
### 1. 先抓演讲目标
- 这是解释型、说服型、汇总型还是答辩型
- 观众最需要带走的一句话是什么
- 第一页最该建立什么预期

### 2. 先做讲述路径
默认先排：
- 问题是什么
- 为什么值得讲
- 核心判断是什么
- 如何展开
- 最后带走什么

不要先堆页面，再回头想怎么讲。

### 3. 每页只做一件事
每页默认只承担一个主旨，并明确：
- 这一页标题是什么
- 这一页想让观众记住什么
- 这是吸引注意、建立判断，还是解释证据

## 工具默认路由
- HTML 演示：优先 `html-ppt`
- 传统 PPT / `.pptx`：优先 `presentations`
- 材料是 PDF 或论文：先用 `pdf` 抽取和理解，再进入演示制作

如果用户提到以下任一需求：
- 演讲
- 逐字稿
- notes
- presenter mode
- 演讲者视图
- 提词器

则在 `html-ppt` 路由下，默认优先选择支持讲者模式的模板和运行时，不要只做观众端页面。

## 模板与风格选择规则
- 不要先从“最炫”的模板开始选。
- 先排除与任务目标冲突的模板。
- 风格服务讲述，不服务自我展示。

默认优先考虑：
- 学术 / 课程 / 研究汇报：克制、清楚、可信
- 小组汇报 / 课堂展示：主判断前置、讲述节奏顺
- 正式分享 / 技术分享：观众视图干净，讲者支持强

## 默认禁忌
不要把演示做成：
- 普通长网页
- 靠滚动阅读的页面
- 首屏像官网 hero section
- 组件很多但逻辑很弱的“高级感”模板
- 文字墙
- 一页多判断并行
- 观众内容和讲者内容混在一起

## notes / speaker script 规则
如果任务适合口头讲述，默认应考虑 notes。

notes 默认用于：
- 提示过渡句
- 备注口头解释
- 标记这一页怎么开场和怎么收

不要把面向讲者的提示直接写在观众页面上。

## 演示类交付前闸门
交付前至少确认：
- 首页是否像演示封面，而不是网页首页
- 是否支持全屏
- 是否支持稳定翻页
- 是否有 presenter mode / notes / 提词支持
- 是否一页一个主旨
- 是否能一眼看出主判断
- 是否存在明显模板味、土味或 AI 平铺感

## 失败纠偏优先顺序
如果演示任务失败，优先从这几类定位问题：
- 路由错：本该走 `html-ppt` / `presentations` 却手搓
- 任务理解错：做成了页面而不是演讲
- 功能缺失：全屏、notes、讲者视图缺失
- 审美翻译错：把抽象风格词错译成流行皮肤
- 排版失稳：一页信息太满、主次不清、节奏混乱
