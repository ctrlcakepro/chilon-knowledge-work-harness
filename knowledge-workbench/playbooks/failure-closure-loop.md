# 失败复盘到永久规则闭环

目标：当一次任务已经做偏、功能缺失、风格跑偏或成品感不足时，不只做当场修正，还要稳定判断“这次偏差该沉淀进哪一层规则”，让同类问题下次更早被拦住。

## 适用场景
默认在以下情况启动：

- 用户明确指出结果做偏了
- 任务已经返工 1 次以上
- 同类偏差已经出现过不止一次
- 当前任务本身就是 harness 校准任务
- 输出虽然能用，但明显暴露了结构性短板

## 闭环目标
一次完整闭环至少回答这 4 个问题：

1. 这次到底错在哪一层
2. 更合适的路径本来应该是什么
3. 这次先怎么修正当前结果
4. 这类问题以后该前置到哪条永久规则

## 默认闭环顺序
1. 先承认核心偏差，不做防御性解释
2. 再判断偏差类型
3. 再切换到更合适的当前路径
4. 最后把偏差沉淀成永久规则或反例

## 偏差类型表
### 1. 路由错
表现：
- 本该走专用 skill，却走成普通聊天
- 本该优先用成熟工具链，却手工重做
- 交付类型和工具能力错配

默认补到：
- `playbooks/tool-routing.md`
- 必要时补 `playbooks/composite-task-decision-table.md`

### 2. 任务识别错
表现：
- 把审稿做成总结
- 把演示做成网页
- 把查询做成泛泛讨论
- 把复合任务做成平均展开

默认补到：
- `playbooks/composite-task-decision-table.md`
- 必要时补 `architecture/layers.md`

### 3. 交付定义错
表现：
- 用户要 PDF / Word / PPT / HTML 成品，却只给文本
- 用户要“可讲”的演示，却只给“可看”的页面
- 用户要“正式评估”，却只给随口点评

默认补到：
- `AGENTS.md` 的交付定义规则
- `playbooks/tool-routing.md`
- 必要时补对应交付协议

### 4. 工作流错
表现：
- 步骤顺序不对
- 该先做判断，结果先铺细节
- 该先定主线，结果先堆内容

默认补到：
- 对应模板文件
- 对应 playbook

### 5. 功能缺失
表现：
- 用户点名的核心功能没做
- 文件能生成但不能用
- 演示能看但不能讲

默认补到：
- `playbooks/tool-routing.md`
- `playbooks/quality-gates.md`
- 必要时补对应专用交付协议

### 6. 评分与验收失灵
表现：
- 明明不过线却被当作可交
- 评分卡选错
- 验收结果说不清

默认补到：
- `architecture/scoring-layer.md`
- `playbooks/scoring-routing.md`
- `playbooks/scoring-output-format.md`

### 7. 审美翻译错
表现：
- 用户给了风格词，但结果只套了流行模板
- “高级感”被做成炫技皮肤
- 信息主次和视觉节奏都失控

默认补到：
- `style/aesthetic-translator.md`
- `style/directions.md`
- 必要时补相应交付协议中的审美禁忌项

### 8. 成品完成度不够
表现：
- 有内容但不像成品
- 结构松散、排版失稳、收束不足
- 只能看，不能直接继续使用

默认补到：
- `playbooks/quality-gates.md`
- `playbooks/scoring-output-format.md`
- 必要时补任务模板

## 永久规则落点表
如果这次偏差需要沉淀，默认优先判断落在哪一层：

- 是“先走错路” -> 补 `tool-routing`
- 是“没认清主任务” -> 补 `composite-task-decision-table`
- 是“验收太松” -> 补 `scoring-layer` / `scoring-routing` / `scoring-output-format`
- 是“审美翻译跑偏” -> 补 `aesthetic-translator` / `directions`
- 是“像半成品” -> 补 `quality-gates`
- 是“属于已知高风险反例” -> 补 `calibration-replays`

不要一出问题就往 `AGENTS.md` 里堆长规则。优先补最贴近问题所在的底层文件。
如果要进一步判断“这条经验到底值不值得升成全局规则”，默认再参考 `knowledge-workbench/playbooks/rule-governance.md`。

## 标准闭环输出格式
### 格式 1：轻量闭环
适用于：
- 单次偏差
- 修正路径已经很明确

推荐结构：
- `偏差类型`：X
- `核心问题`：一句话
- `当前修正路径`：一句话
- `永久规则落点`：补到哪个文件

### 格式 2：标准闭环
适用于：
- 高风险偏差
- 校准任务
- 需要明确后续规则沉淀

推荐结构：
- `偏差归类`：路由错 / 任务识别错 / 功能缺失 / 审美翻译错 / 验收失灵 / 成品完成度不够
- `核心偏差`：一句话承认
- `正确路径`：这次应该怎么做
- `当前修正`：立刻怎么改
- `永久规则`：以后默认拦截点是什么
- `落点文件`：具体补到哪个 playbook / template / style / AGENTS

## 与校准回放的关系
- `calibration-replays.md` 负责保存高价值反例
- 本文件负责判断“这次偏差该怎么沉淀回系统”

默认规则：
- 一次性小偏差，不一定进回放库
- 重复出现、返工成本高、跨任务复现的偏差，应该进回放库

## 进入回放库的判定标准
满足任一项，默认值得写进 `calibration-replays.md`：

- 同类错误已经出现第 2 次
- 一旦走错路，返工成本很高
- 这个偏差会跨多个任务族复现
- 用户已经明确把这类任务当作系统校准样本

## 禁忌
- 不要把复盘写成长道歉
- 不要只说“下次注意”
- 不要只修当前结果，不补系统落点
- 不要一律把规则堆回 `AGENTS.md`
- 不要把局部偶发问题误判成全局永久规则

## 最小执行口径
每次闭环前，至少先回答：

- 这次偏差属于哪一类
- 现在先怎么把当前任务救回来
- 以后该前置到哪份规则
- 这次值不值得写进回放库
