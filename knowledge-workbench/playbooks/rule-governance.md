# 规则治理与瘦身协议

目标：防止 harness 随着使用时长增长而不断堆规则，最终变成笨重、重复、互相打架的系统。

## 核心判断
不是所有经验都值得进入全局规则。

默认分成 4 层：
- `全局规则`：跨任务族、重复出现、代价很高的稳定规律
- `playbook / template / style`：只影响某一类任务的中层规则
- `calibration-replays`：高价值反例与校准样本
- `临时修正`：只修当前任务，不沉淀成长期规则

一句话原则：
- 高频、稳定、跨任务的问题，才值得升到更高层
- 低频、局部、偶发的问题，不要硬塞进全局

## 进入全局规则的门槛
满足大部分条件时，才考虑进入 `AGENTS.md` 或全局架构层：

- 这个问题至少重复出现过 2 次
- 会跨多个任务族复现
- 一旦踩中，返工成本很高
- 不是某个单一交付物的局部习惯
- 能用一句短规则稳定描述

不满足这些条件时，优先放到：
- `playbook`
- `template`
- `style`
- `calibration-replays`

## 不该进入全局的内容
以下内容默认不要直接写进 `AGENTS.md`：

- 某一个具体案例的细节
- 某个单一风格词的局部偏好
- 只对一个工具链生效的操作细节
- 一次性返工经验
- 只能靠很长解释才能成立的规则
- 与已有规则高度重复、只是换了说法的内容

## 新规则落点判断
### 1. 这是跨任务的底层约束吗
是：
- 考虑进 `AGENTS.md`
- 或进 `architecture/`

不是：
- 优先进对应 `playbook / template / style`

### 2. 这是某类任务的固定流程吗
是：
- 进对应 `playbook`

### 3. 这是高频输出结构吗
是：
- 进对应 `template`

### 4. 这是风格翻译或审美禁忌吗
是：
- 进 `style/`

### 5. 这是高价值反例吗
是：
- 进 `calibration-replays.md`

## 规则瘦身动作
当发现规则越来越长时，默认优先做这 5 个动作：

### 1. 合并同类项
如果两条规则本质相同：
- 合并成一条更短、更抽象的规则
- 不保留平行重复表述

### 2. 下沉细节
如果一条规则只影响某类任务：
- 从 `AGENTS.md` 下沉到对应 `playbook / template / style`

### 3. 上提共性
如果多个 playbook 里反复出现同一种底层原则：
- 抽成一条更短的全局规则

### 4. 改“案例描述”为“判断句”
不要写长段案例故事，优先改成：
- 触发条件
- 默认动作
- 禁忌项

### 5. 删除失效规则
如果规则已经：
- 被更高层规则覆盖
- 对当前工作流不再产生判断价值
- 只是历史痕迹

默认应删或合并，不继续留存。

## 规则长度控制
### 1. 全局规则只保留判断句
`AGENTS.md` 里优先放：
- 分流规则
- 优先级规则
- 禁降级规则
- 成品判断规则

不要把工具细节、案例细节、大段解释塞进去。

### 2. 细节一律放到下层文件
需要展开时，默认用链接关系：
- `AGENTS.md` 给一句判断
- 具体细节交给 `playbook / template / style`

### 3. 一条规则最好能回答一个判断
如果一段规则同时在回答 3 件以上事情：
- 说明它太胖了
- 应该拆分或下沉

## 周期性瘦身判断
当出现以下信号时，说明要开始瘦身：

- 同一意思出现了 2 次以上
- 新增规则越来越像补丁
- 只有写很长才能说明白
- 不同文件开始相互重复
- 读规则时需要跨很多段才能做判断

如果需要把这些信号变成正式触发动作，默认再参考 `knowledge-workbench/playbooks/health-check-trigger.md`。

## 默认治理顺序
新增规则前，先问：

1. 这条经验是临时修正，还是长期规律
2. 它是全局问题，还是任务局部问题
3. 它该进 AGENTS、playbook、template、style，还是 calibration
4. 有没有旧规则可以合并或替换
5. 如果不新增全局规则，只补下层文件，够不够

## 最小执行口径
每次准备沉淀规则前，至少先回答：

- 这条规则真的跨任务吗
- 它值得进入全局吗
- 它有没有更合适的下层落点
- 它会不会和已有规则重复
- 加进去之后，是否需要顺手删掉或合并一条旧规则
