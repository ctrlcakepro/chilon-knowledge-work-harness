# 交付前质量闸门

目标：在真正交付前，用最短时间排掉最常见的低级偏差，提升成品感和稳定性。

说明：质量闸门是“查漏补缺”的清单，不等于最终验收。对中等及以上复杂任务、正式文件交付或高风险场景，过完这一层后默认再进入 `architecture/scoring-layer.md` 做过线判断。

## 通用闸门
- 有没有先给结论、主判断或任务标题
- 结构是不是已经像成品，而不是聊天过程
- 用户明确要求的格式、功能、范围有没有漏掉
- 不确定内容有没有被误写成确定事实
- 语言是否清楚，是否还有明显空话、重复或松散段落
- 结尾是否自然收束，并给出可继续推进的下一步

## 阅读总结 / 知识整理
- 有没有抓住主题，而不是只复述细节
- 核心概念之间的关系有没有说清
- 有没有区分“材料原意”和“我的概括”
- 是否已经足够便于继续写作、复习或讨论

## 审稿 / 评估 / 诊断
- 有没有先给明确结论，而不是模糊态度
- 主要问题是否按影响大小排序
- 每个问题后面是否都有可执行修改建议
- 是否贴近该领域的正规评审写法

## 写作 / 论文 / 作业
- 是否先有主论点或中心问题
- 结构是否完整，而不是提纲和正文混杂
- 如果涉及学术内容，概念和论证是否前后一致
- 如果缺数据或出处，是否用稳妥写法处理，而不是编造
- 是否已经接近“可交版本”，而不是只是一版聊天式草稿

## 汇报 / PPT / HTML 演示
- 首页是否一眼能看出主题和主判断
- 一页是否只承担一个主要意思
- 讲述路径是否顺，而不是材料堆叠
- 全屏、翻页、notes、presenter mode 等核心功能是否可用
- 审美上是否克制、有层级、无明显模板味或土味

## 文件交付
- 文件是否真的生成成功
- 文件名是否清楚可辨认
- 用户要求的格式是否一致
- 如果版式重要，是否至少做过一次可视化检查
