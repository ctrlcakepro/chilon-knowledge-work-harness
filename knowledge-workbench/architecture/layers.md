# Harness 分层结构

目标：把现有全局规则从“并列清单”收束成有顺序的执行架构，减少规则增多后出现的优先级混乱。

## Layer 0：可靠性底座
这一层先于所有任务规则。

负责：
- 不编造事实、出处、数据、引文
- 区分材料原文、概括、推理和不确定信息
- 对时效性、出处性、争议性问题优先核对

如果这一层和其他层冲突，默认这一层优先。

## Layer 1：任务识别层
先判断当前请求是什么类型的工作，而不是直接回答。

典型任务：
- 解释 / 总结 / 整理
- 比较
- 审稿 / 评估 / 诊断
- 写作
- 修改
- 查询
- 设计与演示
- 文件交付

这一层回答的是：
- 现在到底在做什么任务
- 当前用户最需要哪种产出

## Layer 2：交付定义层
在开始做之前，先定义最后要交什么。

常见交付物：
- 简短结论
- 提纲
- 表格
- 完整成稿
- PDF
- Word / Docx
- 表格文件
- PPT / Slides
- HTML 演示

这一层回答的是：
- 最终产物是什么
- 用户要求的是“建议”，还是“可直接使用的成品”

## Layer 3：路由层
先选正确 skill 或工具链，再进入内容生成。

优先级通常为：
1. 现成专用 skill
2. 成熟运行时 / 已安装工具链
3. 原生简单实现
4. 手工从零实现

这一层回答的是：
- 这项任务该走什么能力
- 要不要避免降级
- 路由完成后，主评分卡应该先选哪一张

## Layer 4：工作流层
确定路由之后，再进入任务对应的模板、协议和 playbook。

例如：
- 阅读总结模板
- 审稿模板
- 论文提纲模板
- 演示交付协议
- 校准回放

这一层回答的是：
- 当前任务按什么流程推进
- 先做哪一步，再做哪一步

## Layer 5：表达与审美层
在主路径已经明确后，再处理呈现问题。

负责：
- 语气
- 结构层次
- 风格翻译
- 版式气质
- 信息密度和节奏

这一层不能反过来主导任务本身。
也就是说，不能因为想“更高级”就破坏交付物定义和主任务逻辑。

## Layer 6：质控层
在交付前统一做自检。

负责：
- 结论是否前置
- 结构是否像成品
- 功能是否齐全
- 排版是否失稳
- 是否遗漏用户点名的要求

这一层的作用是阻止“看上去差不多就交”的低稳定性输出。

## Layer 7：评分层
在质控之后，把结果从“自检通过”推进到“是否过线”的明确判断。

负责：
- 选出当前任务最关键的 4 到 5 个评分维度
- 判断这版到底能不能交
- 决定是否返工，以及返工优先修哪里
- 区分“可交一版”和“稳妥终版”

这一层的作用是阻止“感觉还行就收尾”的主观交付。

## Layer 8：校准层
如果任务属于高风险重复场景，或者以前已经做偏过，再进入反例和纠偏回放。

负责：
- 命中已知失败案例
- 提前发现重复偏差
- 启动永久规则
- 判断这次偏差该沉淀回哪一层规则

这一层不替代主工作流，而是作为风险前哨。

## 默认执行顺序
1. 可靠性底座
2. 任务识别
3. 交付定义
4. 路由
5. 工作流
6. 表达与审美
7. 质控
8. 评分
9. 校准复盘（高风险场景前置启用）

## 一句话理解
先判断“这是什么任务、最后交什么、该走什么工具”，再决定“写成什么样、做得多好看”，最后再判断“这版到底有没有过线”。
