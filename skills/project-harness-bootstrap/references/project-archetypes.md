# Project Archetypes

Use this file only when the project description is broad, mixed, or under-specified.

Pick one primary archetype and, if needed, one secondary archetype.

## 1. Knowledge / Research

Use when the project mainly handles:

- reading
- note-making
- research planning
- paper or assignment writing
- review or diagnosis
- presentation preparation

Emphasize in project `AGENTS.md`:

- source handling
- evidence vs inference
- repeatable reading / writing / review workflows
- output forms such as outlines, drafts, reviews, notes, and decks

## 2. Build / Engineering

Use when the project mainly handles:

- coding
- debugging
- refactoring
- testing
- automation
- repository maintenance

Emphasize in project `AGENTS.md`:

- repo layout
- test and verification commands
- change boundaries
- review expectations
- environment and tooling preferences

## 3. Content / Writing

Use when the project mainly handles:

- article production
- scripts
- newsletters
- copy
- editorial packaging

Emphasize in project `AGENTS.md`:

- voice and audience
- content pipeline
- revision workflow
- output formats
- approval and publishing boundaries

## 4. Design / Presentation

Use when the project mainly handles:

- deck creation
- visual direction
- demo pages
- presentation packaging
- speaker support

Emphasize in project `AGENTS.md`:

- audience and communication goal
- design direction
- presentation workflow
- file output expectations
- functional presentation requirements such as fullscreen, notes, presenter view

## 5. Operations / Coordination

Use when the project mainly handles:

- planning
- task coordination
- document upkeep
- process management
- cross-file organization

Emphasize in project `AGENTS.md`:

- status tracking
- task intake and triage
- file conventions
- decision logging
- handoff protocol

## Mixed projects

For mixed projects:

1. Pick the archetype that defines the main deliverable.
2. Use the secondary archetype only to add support workflows.
3. Do not average all archetypes together.

Example:

- A research project that ends in a deck: primary `Knowledge / Research`, secondary `Design / Presentation`
- A coding project with heavy docs: primary `Build / Engineering`, secondary `Content / Writing`

## What belongs where

Put these into project `AGENTS.md`:

- durable project workflows
- project-specific output expectations
- file and path conventions
- project-specific confirmation rules

Put these into `PROJECT-WORKFLOW.md`:

- quickstart map
- task intake guide
- common request shapes
- main deliverable list
- practical workflow overview
